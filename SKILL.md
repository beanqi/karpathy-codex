# Karpathy Codex


## 核心原则

### 1. 先需求，再设计，再编码

当任务并非完全明确时，不要立刻开始编码。
先澄清真正需要的结果，不断与用户沟通需求，直到你完全理解。
再选择能满足目标的最小设计。
最后再实现。

### 2. 先思考，再动手

在提出改动前先阅读相关文件。
识别可能的根因、局部约束，以及最小且合理的介入点。
对于非平凡任务，修改前先给出简短计划。

### 3. 优先简单方案

选择能完整解决当前请求的最简单方案。
优先复用现有模式、组件、辅助函数与约定，而不是新增抽象。
只有在重复或复杂度已经明显需要时，才引入新抽象。

### 4. 做外科手术式改动

在正确解决问题的前提下，让 diff 尽可能小。
避免无关清理、大范围重命名、格式抖动与猜测性重写。
除非用户明确要求重构，否则保持现有可运行架构。

### 5. 目标驱动

每一处代码改动都要与用户目标直接关联。
除非为保证正确性所必需，否则不要扩展成“顺手优化”。
存在权衡时，优先选择最易评审、验证和维护的方案。

### 6. 为关键代码元素添加中文注释

在编写或修改代码时，请在关键函数、结构体与重要字段上方添加简洁中文注释。
除非用户或仓库明确要求其他语言，否则这是本 skill 的默认文档要求。

注释范围：

- 关键函数：说明目的、核心行为和重要副作用
- 关键结构体/类型：说明职责和主要使用语境
- 重要字段：在不明显时说明含义、单位/范围或生命周期约束

注释位置与风格：

- 注释放在目标函数、结构体/类型或字段的正上方
- 使用简洁中文，避免重复名称已表达的信息
- 优先说明意图与约束，而非逐行复述
- 行为变化时同步更新注释
- 对于简单局部变量或显而易见的单行逻辑，可不写注释

## 默认工作流

### 阶段 1：对齐需求

首先与用户对齐需求，直到你完全理解他们想要什么，以及为什么想要它。

### 阶段 2：讨论最小可行设计

当需求足够清晰后，在改代码前先描述实现路径。

对于非平凡任务，应包含：

- 可能的根因或实现路径
- 可能变更的文件或层级
- 能解决问题的最小设计
- 至少一个备选方案及其当前不优先的原因
- 明确权衡（复杂度、风险、交付速度、可维护性）
- 为什么更大范围方案当前没有必要

将设计拆成可快速评审的小块呈现。
除非用户明确要求，不要一次性输出巨大的规格文档。

### 阶段 3：把设计转成最小实现计划

方案可接受后，把它转成紧凑的执行计划。

好的计划应当：

- 将工作拆成少量具体步骤
- 在已知时指明可能涉及的文件或模块
- 包含验证或测试步骤
- 细节足够执行，但不过度膨胀成项目管理表演

优先小步、可评审的推进，而非冗长且推测性的路线图。

如果用户只要设计，请在需求与设计输出后停止。
在该模式下不要强行改代码；给出包含权衡、假设与清晰下一步的简洁建议即可。

### 阶段 4：谨慎实现

只修改任务所需文件。
在创建新辅助逻辑前优先复用现有工具。
除非需求要求变更，否则保持函数签名与公共接口稳定。
注释与文档若仍准确则保留；行为变化时及时更新。
对新增或修改的关键函数、结构体/类型及重要字段，在上方补充简洁中文注释。

### 阶段 5：验证与汇报

执行或说明最小且有意义的验证。
结束时总结：

- 改了什么
- 为什么改
- 如何验证
- 剩余风险、假设与后续工作

## 压缩规则

不要为小任务强行套用冗长的“需求-设计-计划”流程。
若请求已足够具体、低风险且局部，可压缩为：

1. 简短任务界定
2. 简短设计选择
3. 实现与验证

目标是有意识地执行，而不是流程官僚化。

## 需要避免的反模式

### 不要过度设计

除非当前任务明确需要，否则不要引入新架构、插件系统、服务层或通用框架。
不要为假想的未来功能提前优化。
不要“以防万一”创建扩展点。

### 不要过度抽象

不要仅为“看起来更干净”而提取辅助函数、基类、hooks、工具层或共享层。
只有当重复逻辑或现有复杂度确实让局部代码明显变差时，才进行抽象。
与过早间接层相比，适度重复往往更可取。

### 不要过度防御或过度填充

除非满足以下条件，否则不要添加兜底分支、配置开关、额外守卫、重试、功能开关或超防御处理：

- 当前需求明确要求
- 该区域在仓库中已有此类约定
- 有具体证据表明该故障模式确实存在

避免写出主要用于“心理安慰”而非解决任务的代码。

### 不要把规划变成表演

短计划足够时，不要产出超大规格、穷举方案矩阵或冗长实施方案。
没有真实需要时，不要发明额外阶段、子项目或后续工单。

## 任务类型指引

### 缺陷修复

打补丁前先还原故障模式。
将问题定位到最窄且最可能的原因。
除非有证据支持，否则避免跨多层“散弹式修复”。
若无法精确复现，应说明修复所依据的假设。

### 功能开发

先把功能嵌入现有架构，再考虑新架构。
优先对现有模块、路由、组件与状态流做增量扩展。
仅新增该功能所需的最小 API 面。

### 重构

仅在重构能直接支持目标结果或显著降低风险时进行。
优先提炼、重命名与简化，而非结构性重写。
除非用户要求改变行为，否则保持行为不变。

### 代码评审

优先关注正确性、安全性、可靠性与可维护性。
先指出最高信号的问题。
优先给出具体且最小的建议，而非泛泛风格评价。
当改动可接受时，明确说明“可通过”。

### 测试工作

添加能证明行为变化的最小测试。
复用现有测试风格、夹具与辅助工具。
除非仓库已采用或用户明确要求，否则不要引入大型新测试框架。

### UI 与设计实现

将设计工作视为受现有产品约束的实现任务。
在发明新布局或交互规则前，先对齐邻近已有模式。
优先使用现有设计 token、间距尺度、排版体系与共享组件。
当收到截图、规格或设计说明时：

- 识别最关键且最小的可见差异
- 映射到当前组件结构
- 以最小改动弥合差距

不要另起炉灶创造新设计系统。
不要为“视觉一致性”重做无关页面样式。
若关键设计上下文缺失，应保守推进并明确假设。

## 输出风格

对于非平凡任务，通常按以下顺序输出：

1. 需求摘要或假设
2. 最小可行设计
3. 最小实现计划
4. 实现与验证结果

各部分保持紧凑。
优先使用清晰直白的语言，少用术语堆砌。

## 禁止事项

在阅读相关文件前不要开始编码。
对于非平凡任务，不要跳过需求与设计讨论。
不要默认引入新抽象。
不要在同一 diff 中混入无关清理。
当证据不完整时，不要假装确定。
不要隐瞒风险、假设或未完成的验证。
